Redis используется для быстрых временных данных.

Хранит:

# 8. Как проходит бой

## 1. Игрок нажимает “Бой”

Frontend отправляет REST-запрос:

```text
POST /api/battles/start
```

Backend:

* ищет соперника;
* создает battleId;
* сохраняет начальное состояние в Redis;
* возвращает battleId.

---

## 2. Игрок подключается к SignalR

Frontend подключается:

```text
/battleHub
```

и вызывает:

```text
JoinBattle(battleId)
```

Backend добавляет игрока в группу SignalR:

```text
battle:{battleId}
```

---

## 3. Бой начинается

Backend отправляет событие:

```text
BattleStarted
```

На фронте отображается экран боя.

---

## 4. Игрок выбирает персонажей

Варианты:

### Один персонаж

Игрок нажимает на своего персонажа.

Затем нажимает на врага.

Frontend отправляет:

```text
SelectTarget
```

### Группа персонажей

Игрок выделяет областью несколько своих персонажей.

Затем нажимает на врага.

Frontend отправляет:

```text
SelectGroupTarget
```

---

## 5. Backend обрабатывает команду

Backend:

* проверяет battleId;
* проверяет игрока;
* проверяет, что персонажи принадлежат игроку;
* проверяет, что цель существует;
* обновляет состояние боя в Redis;
* рассылает событие через SignalR.

---

## 6. Игровой цикл боя

На backend работает Battle Engine.

Каждые условные 200–500 мс он:

* читает состояние боя из Redis;
* рассчитывает действия персонажей;
* применяет урон;
* обновляет cooldown;
* проверяет смерть персонажей;
* проверяет окончание боя;
* отправляет изменения через SignalR.

---

# 9. Почему Redis, а не PostgreSQL для боя

PostgreSQL используется для надежного хранения.

Redis используется для активных боев, потому что:

* быстрее;
* хранит временное состояние;
* не нагружает базу частыми обновлениями;
* подходит для realtime-сценариев.

После завершения боя результат сохраняется в PostgreSQL.

---

# 10. Завершение боя

Когда бой закончен:

Backend:

1. Определяет победителя.
2. Считает награды.
3. Сохраняет результат в PostgreSQL.
4. Обновляет рейтинг.
5. Выдает сундуки.
6. Удаляет временное состояние из Redis.
7. Отправляет игрокам событие:

```text
BattleFinished
```

# 11. Минимальная архитектура MVP

Для первой версии можно сделать проще.

## MVP содержит:

* Telegram login;
* профиль игрока;
* стартовый огород;
* 3 базовые культуры;
* обычный PvP;
* бой 3 на 3;
* SignalR-синхронизацию;
* Redis для состояния боя;
* PostgreSQL для хранения результата.

## Пока можно не делать:

* рынок;
* сложные турниры;
* обмен предметами;
* редкие теплицы;
* сложную экономику;
* много типов сундуков.

# 13. Главные паттерны

## Clean Architecture

Разделение проекта на слои.

## Service Layer

Отдельные сервисы для бизнес-логики.

## Repository

Для работы с PostgreSQL.

## Redis State Store

Для хранения активных боев.

## Event-driven внутри боя

Battle Engine генерирует события:

* damage;
* heal;
* ability ready;
* ability used;
* defeated;
* finished.

## State Machine

Бой имеет состояния:

```text
Created
WaitingForPlayers
Started
InProgress
Finished
Cancelled
```

---

# 14. Главный принцип

PostgreSQL хранит историю и постоянные данные.

Redis хранит активный бой.

SignalR доставляет изменения игрокам.

.NET рассчитывает бой и остается источником истины.

Frontend только отображает бой и отправляет команды игрока.
